# 剧本 YAML Schema 设计文档

> 本文定义「AI 小说转剧本工具」输出的结构化剧本 YAML Schema，并逐项说明设计原因。
> Schema 的**单一事实源**是 Java record（见 `src/main/java/com/novel2screenplay/model/`）——
> record 即 Schema，本文档照 record 编写，二者必然一致。

---

## 1. 为什么用 YAML

剧本天然是**层级化**的结构（剧本 → 场景 → 动作/对白 → 台词要素），需要一种既能表达层级、又便于人类阅读与修改的格式。对比候选：

| 格式 | 可读性 | 可手改 | diff 友好 | 层级表达 | 结论 |
|---|---|---|---|---|---|
| JSON | 一般（括号/引号噪声多） | 易出错（漏逗号） | 一般 | 好 | 适合机器，不适合作者手改 |
| XML | 差（标签冗长） | 繁琐 | 差 | 好 | 过重 |
| **YAML** | **好** | **好** | **好** | **好** | ✅ 采用 |

选择 YAML 的核心理由：

1. **人类可读、可手改**：作者拿到的是"初稿"，必然要继续打磨。YAML 缩进即层级、无闭合标签噪声，作者可直接用文本编辑器修改，无需懂编程。
2. **diff 友好**：YAML 按行组织，改一句台词只产生一行 diff，天然适配 git 等版本管理，便于追踪修改、协作评审。
3. **贴合剧本层级**：剧本的"场景—动作—对白"嵌套，用 YAML 的 map/list 嵌套表达直观自然。
4. **强类型保证合法**：本工具用 Java record 序列化为 YAML，输出**必定是合法 YAML 且字段类型受控**，不会出现模型直接吐 YAML 时的缩进错乱、引号未闭合等问题。

## 2. 顶层结构 Screenplay

```yaml
meta:                          # 溯源信息（改编自几章、共几场）
  source_chapters: 3
  scene_count: 12
title: 铁面青衫                 # 剧名
logline: 剑客沈砚追查黑风寨…     # 一句话故事梗概
style: 剧集                     # 改编风格（默认主流：剧集）
characters: [...]              # 全局人物登记表
episodes: [...]                # 分集编排层（剧集形态填充；单本形态省略）
scenes: [...]                  # 场景列表
```

| 字段 | 类型 | 说明 | 设计原因 |
|---|---|---|---|
| `meta` | ScreenplayMeta | 溯源信息（`source_chapters` 改编自几章、`scene_count` 共几场） | 让作者一眼看清这份初稿的来历与规模；由程序计算填充，不靠模型 |
| `title` | string | 剧名 | 剧本基本元信息 |
| `logline` | string | 一句话故事梗概 | 行业惯例（logline 是剧本推介的标准要素），点明主角/目标/冲突，便于作者与读者快速把握全局 |
| `style` | string | 改编风格——默认主流**剧集**；电影/话剧/短剧/分镜为单本形态 | 同一小说可改编为不同体裁；剧集是网文改编主航道，故为默认 |
| `characters` | Character[] | 全局人物登记表 | **抽成独立表而非散落各场景**，见 §3 |
| `episodes` | Episode[] | 分集编排层（剧集形态填充，单本形态省略） | **集≠章**，叠加在 scenes 之上而非重构 Schema，见 §7.6 |
| `scenes` | Scene[] | 场景列表 | 剧本主体 |

### 为什么把 characters 抽成独立登记表

如果人物信息散落在每个场景里，会带来三个问题：**重复**（同一角色在多场景重复定义）、**不一致**（跨场景/跨章称呼漂移，如"李白/太白/谪仙人"被当成三个人）、**难复用**。

独立登记表解决了这些：

- **去重**：每个角色只定义一次。
- **跨场景/跨章一致**：登记表是人名归一的单一依据；本工具在抽取每个场景时都会把登记表注入提示词，保证全篇称呼统一（这是质量的命门）。
- **可复用**：别名表（`aliases`）让"同一人物的不同称呼"显式可查。

## 3. 人物 Character

```yaml
- name: 铁面人
  aliases:
  - 黑风寨的人
  - 苏窈的兄长
  description: 身覆半张铁面具、怀揣青铜令牌的神秘黑影…
```

| 字段 | 类型 | 说明 | 设计原因 |
|---|---|---|---|
| `name` | string | 主名（规范称呼） | 全篇对白的 `character` 字段都引用此主名，保证一致 |
| `aliases` | string[] | 别名/称呼列表 | 收录绰号、尊称、自称等；是跨章人名归一的依据。例：第3章揭示"铁面人"即"苏窈的兄长"，会被合并进同一条目 |
| `description` | string | 一句话人物设定 | 给作者快速画像，也为后续场景生成提供人设上下文 |

## 4. 场景 Scene

```yaml
- id: S1
  heading:
    int_ext: 内
    location: 听雨楼客栈大堂
    time_of_day: 黄昏
  summary: 沈砚独坐角落，青衫湿透，端酒未饮。
  action:
  - 暮色沉沉，雨声淅沥。听雨楼客栈大堂内人声嘈杂…
  dialogue:
  - character: 苏窈
    parenthetical: 清亮
    line: 客官，借个座可好？
  transition: 切至
  source:
    chapter: 1
    excerpt: 暮色四合，临川镇外的"听雨楼"客栈里挤满了避雨的旅人…
```

| 字段 | 类型 | 说明 | 设计原因 |
|---|---|---|---|
| `id` | string | 全局唯一场景号（S1、S2…） | **见下**：可编辑、可重排、可引用 |
| `heading` | Heading | 场景标题（slugline） | 对标行业剧本 slugline，见 §5 |
| `summary` | string | 本场一句话概要 | **非严格行业字段，但极实用**：让作者不必读完整场就能把握全剧节奏，也便于大纲视图 |
| `action` | string[] | 可拍摄的动作/场景描述段落 | 对标剧本 action；用列表而非长字符串，便于逐段编辑、增删 |
| `dialogue` | DialogueLine[] | 对白列表 | 见 §6 |
| `transition` | string | 转场（切至/淡出/闪回），可空 | 对标剧本 transition；多数场景无需转场，故可空 |
| `source` | SourceRef | 来源出处 | **本工具特色字段（可信度/可追溯）**，见 §7 |
| `craft` | SceneCraft | 编剧笔记（戏剧注释层） | **让"好剧本"可见、可校验**，见 §7.5 |

### 为什么给 scene 加全局 id

- **可编辑/可重排**：作者可能调整场景顺序、删减、合并。稳定的 id 让"引用某一场"成为可能，也便于工具链后续按 id 定位修改。
- **统一编号、消除模型不一致**：模型自行编号常出现重复或从 1 重启。本工具在组装阶段由程序统一重排为 `S1..Sn`，保证全局唯一、连续。

## 5. 场景标题 Heading（slugline）

```yaml
heading:
  int_ext: 内          # 内外景
  location: 听雨楼客栈大堂
  time_of_day: 黄昏
```

对标影视行业 slugline 三要素（中文剧本写作「内 听雨楼客栈大堂 黄昏」）：

| 字段 | 类型 | 取值 | 设计原因 |
|---|---|---|---|
| `int_ext` | enum | 中文 `内` / `外` / `内外` | **用枚举而非自由字符串**，取值受控、可校验；序列化为中文与全篇一致、便于作者手改；导出 Fountain 时再映射回行业标准 INT./EXT.（底层枚举常量不变，国际兼容不丢） |
| `location` | string | 中文具体地点 | 场景切分的核心依据之一（地点变化即换场） |
| `time_of_day` | string | 中文：清晨/白天/黄昏/夜晚/午夜/黎明 | 场景切分的另一依据（时间跳转即换场）；约定用中文，避免中英混杂 |

> **场景切分规则**：本工具按「地点或时间发生切换」切分场景——这正是 slugline 的工业逻辑，一章通常切出多个场景。

## 6. 对白 DialogueLine

```yaml
dialogue:
- character: 沈砚
  parenthetical: 淡淡
  line: 请便。
- character: 旁白
  line: 雨夜的客栈，是故事的起点。
  type: 旁白          # 仅旁白/画外/内心时标注
```

对标行业剧本的「角色 + 提示 + 台词」三件套，并支持区分对白类型：

| 字段 | 类型 | 说明 | 设计原因 |
|---|---|---|---|
| `character` | string | 说话角色（**必须是 characters 登记表中的主名或别名**） | 保证对白归属与人物表一致；本工具校验环节会检查越界角色 |
| `parenthetical` | string | 括号提示（语气/小动作），可空 | 对标剧本 parenthetical，如「(冷笑)」；可空 |
| `type` | enum | 对白类型：`旁白` / `画外` / `内心`，普通对白留空 | 真实剧本常区分旁白、画外音(O.S.)、内心独白(V.O.)；留空即普通同期声，保持 YAML 简洁；导出 Fountain 时映射为 (V.O.)/(O.S.) |
| `line` | string | 台词 | 对白正文，不应为空 |

## 7. 来源出处 SourceRef（本工具特色 / 可信度设计）

```yaml
source:
  chapter: 1
  excerpt: 暮色四合，临川镇外的"听雨楼"客栈里挤满了避雨的旅人…
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `chapter` | int | 该场景改编自原文的第几章（由程序回填，可靠） |
| `excerpt` | string | 触发该场景的原文片段摘录 |

**为什么要有 source（设计动机）**：AI 生成的剧本常被质疑是"黑盒"——作者不知道某个场景是凭空发挥还是忠于原文，因而不敢信任、不便校对。`source` 让**每个场景都可追溯回原文出处**，作者可逐场对照，确认改编是否忠实。这把"不可信的自动生成"变成"可验证的辅助初稿"，是本工具的核心价值主张之一。导出 Fountain 时，出处以笔记 `[[ ]]` 形式保留，可追溯性不丢失。

## 7.5 编剧笔记 SceneCraft（让"好剧本"可见、可校验）

```yaml
craft:
  objective: 苏窈需要一个干燥座位安顿自己并观察环境。
  conflict: 沈砚的沉默与按剑的手形成无形屏障，暗示拒绝靠近。
  turn: 沈砚抬眼确认面容后默许，伞尖水珠落地声成为两人第一个共享的节奏。
  function: SETUP
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `objective` | string | 本场主驱动角色想要什么（戏剧目标） |
| `conflict` | string | 阻碍该目标的对抗力量 |
| `turn` | string | 本场结束时发生的变化（情势/关系/情绪/信息的翻转）——不能是"无变化" |
| `function` | enum | 主要戏剧职能：`ADVANCE_PLOT`(推进剧情) / `REVEAL_CHARACTER`(塑造人物) / `ESCALATE_CONFLICT`(升级冲突) / `SETUP`(铺垫) / `PAYOFF`(回收) |

**为什么要有 craft（设计动机）**：剧本"格式正确"不等于"写得好"。好剧本的内核是**戏剧craft**——每场戏有目标、有冲突、有转折。一个常见失败是 AI 只把小说段落重排成 slugline+对白，场景平铺直叙、没有戏剧张力。

我们刻意**把可拍摄的剧本正文与"编剧笔记"分离**：

- 真实剧本里并没有"冲突:"这种字段——那是编剧的工作笔记，不是被拍摄的内容。把它强行写进正文会污染剧本。
- 但把它**显式建模为独立注释层**，有三重价值：① 逼迫模型在生成时进行戏剧思考（而非机械转写）；② 给作者一份"这场为什么存在"的判断依据，`turn` 为空即提示该场是可合并的死场景；③ **可校验**——校验器检查每场 craft 完整性并触发自检修复闭环。

配套的两条"硬约束"也由 prompt + 校验闭环共同保证：

- **展示而非讲述**：`action` 只写镜头看得见、麦克风听得见的内容；校验器扫描 action 中的心理描写标记（"心想/意识到/暗暗…"），命中即要求外化为可拍摄画面。
- **对白去信息倾倒**：禁止用长段对白硬塞背景；校验器对超长单句对白告警，触发拆短或外化。

> 这一层是本工具"理解什么是好剧本"的直接证据——它不只转换格式，还应用戏剧原则并能自检。

## 7.6 分集 Episode（剧集形态：集 ≠ 章）

```yaml
episodes:
- number: 1
  title: 雨夜借座
  synopsis: 苏窈冒雨入客栈寻人，与沈砚初遇，亮出"黑风寨"线索。
  hook: 沈砚仰头饮尽那碗一直未动的浊酒——他为何对"黑风寨"如此在意？
  scene_ids: [S1, S2, S3]
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `number` | int | 集号，从 1 递增 |
| `title` | string | 集名 |
| `synopsis` | string | 本集一句话梗概 |
| `hook` | string | **集尾钩子**：本集结尾的悬念/转折/危机，驱动观众追看下一集 |
| `scene_ids` | string[] | 本集包含的场景 id（引用 `scenes` 中的 id，按播出顺序） |

**为什么要有"集"这一层（设计动机）**：小说（尤其网文）改编的主航道是**剧集**（电视剧/网剧），不是电影。而剧集的核心结构单元是"**集**"——它和小说的"章"不是一回事：

- **集是创作决策，不是机械切割章节**：电视剧从分集大纲就开始按"集"创作，每集是一个相对完整的叙事单元（有起承转合 + 集尾钩子）。一集可能横跨多个小说章，也可能把一章拆成多集。
- 因此分集不能等同于章节切分，必须由"懂节奏"的一步专门完成——本工具用 `EpisodePlanner` 读完整场景清单、模拟编剧写分集大纲（可 `?episodes=N` 指定集数，或由模型按节奏自动决定）。

**为什么用"叠加引用层"而非重构 Schema**：`episodes` 不持有场景内容，只通过 `scene_ids` **引用** `scenes` 里的场景。这样：

- `scenes` 仍是全局扁平、统一编号的"素材库"，既有的溯源（`source`）、编剧笔记（`craft`）、校验、导出全部原样复用，零破坏。
- 集只是"如何编排"的一层视图；**单本形态（电影/话剧/短剧/分镜）下 `episodes` 省略**，向后完全兼容。
- 这正是"集≠章"在数据结构上的体现：**内容（场景）与编排（分集）解耦**。

**可校验**：`EpisodeValidator` 检查分集是否**覆盖全部场景、不重不漏、不引用不存在的场景**，以及每集是否有集尾钩子（剧集命门）；集层问题以告警计入 `X-Validation-Warnings`。

## 8. 预留扩展位（本期不实现，仅论证演进方向）

Schema 刻意保持精简，但为常见的进阶剧本要素预留了清晰的扩展路径：

| 扩展 | 形态 | 用途 |
|---|---|---|
| 双人对白 dual dialogue | `DialogueLine.dual: bool` | 两角色同时说话（剧本左右分栏） |
| 分镜 shot | `Scene.shots: Shot[]` | 在场景内进一步细分镜头（景别、机位、运动） |

> 注：对白类型（旁白/画外/内心，对标 V.O./O.S.）原为扩展位，现已实现为 `DialogueLine.type`，见 §6。

之所以暂不实现上面两项：遵循"简洁优先"，避免为低频场景过度设计；但由于 Schema 是强类型 record，新增字段是**向后兼容的纯增量**（旧 YAML 仍合法），演进成本低——`type` 的落地正是这条演进路径的范例。

## 9. record = Schema 单一事实源

本 Schema 不是一份独立维护的规范文档，而是**直接由 Java record 定义**：

- 抽取：Spring AI 的结构化输出分析 record → 生成 JSON Schema 注入提示词 → 把模型返回反序列化为 record 对象。
- 输出：Jackson `YAMLMapper` 把 record 序列化为 YAML（蛇形命名、省略空字段）。
- 文档：本文照 record 编写。

三者共用同一套 record，从根本上杜绝了"文档与实现不一致"。字段命名约定：Java 用驼峰（`intExt`、`timeOfDay`），YAML 自动转蛇形（`int_ext`、`time_of_day`）。
